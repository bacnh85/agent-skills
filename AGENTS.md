# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project Overview

`@bacnh85/agent-skills` is a Node.js/TypeScript CLI for managing AI agent skills from git or local sources. It can add/update/remove skills in this repository and install/uninstall them into project or user `.agents/skills/` directories.

## Tech Stack

- Runtime: Node.js >= 20
- Language: TypeScript, ESM (`"type": "module"`)
- Module settings: `NodeNext` module and resolution
- Tests: built-in `node:test` against compiled files in `dist/test/*.test.js`
- CLI entrypoint: `src/cli.ts`, compiled to `dist/src/cli.js`

## Common Commands

```bash
npm run typecheck
npm run build
npm test
```

Notes:

- `npm run build` removes `dist/`, compiles TypeScript, and chmods the compiled CLI.
- `npm test` runs `npm run build` first, then executes compiled tests.
- Do not edit `dist/` directly; it is generated by the build.

## Repository Layout

```text
src/                 TypeScript source for the CLI and library helpers
test/                TypeScript node:test suites
skills/              Curated skill directories managed by this project
.agents/skills/      Locally installed skills for agent use
skill-registry.json  Registry of curated skills in this repository
skill-history.jsonl  Append-only skill operation history
skills-lock.json     Skill install lock data
```

## Coding Guidelines

- Preserve strict TypeScript and ESM style; include `.js` extensions in relative imports.
- Keep public CLI behavior stable unless explicitly asked to change it.
- Prefer small, focused changes and reuse existing helpers in `src/config.ts`, `src/discovery.ts`, `src/installer.ts`, `src/manager.ts`, `src/registry.ts`, `src/skill.ts`, `src/ui.ts`, and `src/version.ts`.
- Keep CLI parsing helpers import-safe; tests import `src/cli.ts` symbols and should not execute the command as a side effect.
- For filesystem mutations, follow the existing staging/transaction/rollback patterns and validate destination paths before writing.
- When creating temporary git repositories in tests, set `commit.gpgsign=false` so developer signing configuration does not break fixtures.
- Do not print, commit, or modify secrets such as `.env.local` values.

## Skill Authoring Guidelines

Skill directories should use this structure:

```text
skills/{skill-name}/
  SKILL.md              Required skill definition
  scripts/              Optional executable scripts
  references/           Optional supporting docs loaded on demand
  lib/                  Optional shared script code
```

- Skill directory names are kebab-case.
- `SKILL.md` must use this exact uppercase filename.
- Keep `SKILL.md` concise and put large reference material in separate files.
- Prefer scripts for repeatable workflows; scripts should write status to stderr and machine-readable output to stdout when practical.

## Platform Lessons

### Spawning `npm` on Windows

- On Windows, `npm` is installed as command shims. Node's `child_process`
  (`execFileSync`/`spawnSync`) cannot resolve `npm` directly in some shells
  (`ENOENT`), and executing `npm.cmd` directly can fail with `EINVAL`.
- Prefer routing Windows npm calls through `cmd.exe /d /s /c` via
  `createNpmInvocation()`, while using direct `npm` execution on POSIX. Avoid
  `shell: true` with an args array because Node emits `DEP0190` warnings.
- When a check fails, surface the underlying reason (e.g. `result.error.message`)
  instead of a bare "Unable to ..." so future platform issues are diagnosable.
- Detection signal: a feature works on macOS/Linux but fails on Windows with a
  silent `ENOENT`-style catch — suspect a missing `.cmd` shim for `npm`/`npx`.

## Verification Expectations

- For source changes, run `npm run typecheck` or `npm test` depending on scope.
- For CLI behavior changes, add/update `test/cli.test.ts` or the nearest relevant test and run `npm test`.
- For registry/skill management changes, verify both success and rollback/error paths when practical.
- Tests assume POSIX path separators and unprivileged symlinks; on Windows, some
  integration tests fail on `\` vs `/` and `EPERM` symlink creation. These are
  pre-existing platform issues, not regressions from version/upgrade work.
